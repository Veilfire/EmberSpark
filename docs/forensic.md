# Forensic Review

EmberSpark can optionally record the complete prompt → reasoning →
tool → memory chain for a task run, encrypt it at rest with a
per-run age identity, and surface it as an admin-gated viewer in the
web UI (and a `spark forensic` CLI). This page is the source-level
reference. For operator walkthroughs see the wiki's
[Forensic Review Guide](../wiki/Forensic-Review-Guide.md).

## Goals

1. **Debuggability** — reconstruct exactly what a run saw, decided,
   and did, long after the fact.
2. **Accountability** — the immutable record is gated on admin role
   and every read is audited.
3. **Bounded risk** — captures are opt-in per run, per-run encrypted,
   TTL-expiring, and wipeable with a single key-delete.
4. **Zero overhead when off** — the default is off. A run with
   `forensic.enabled: false` performs no extra DB writes and never
   creates an age identity.

## Data model

Two SQLite tables (see `spark/persistence/learning_models.py`):

```
forensic_captures
  run_id         PK    task run id
  agent_name         
  task_name
  enabled_by         who turned it on
  enabled_reason     required free-text justification
  captured_at
  expires_at         captured_at + ttl_hours
  vault_key          secrets-provider key holding the per-run age identity
  iteration_count    engine iteration count
  snapshot_count
  wiped_at           non-null once shredded

forensic_snapshots
  id                  PK
  run_id              FK → forensic_captures.run_id
  iteration           engine iteration (1-indexed)
  sequence            within-iteration ordering
  span_id             optional pointer into run_spans (F4)
  kind                prompt | model | tool | memory_retrieved
                     | memory_written | reflection
  captured_at
  payload_encrypted   BLOB — age-encrypted JSON
```

The Pydantic schemas for each payload kind live in
`spark/forensic/schemas.py`.

## Encryption

Each run gets its **own** X25519 identity generated by
`ForensicWriter.start()`. The private key is stored via the
:class:`SecretManager` (i.e. the age vault) under the key
``forensic:<run_id>``, and the public recipient is held in memory
for the duration of the run. Every snapshot is encrypted to the
recipient before touching SQLite.

Wiping a capture deletes the vault key first — that is the
cryptographic-shred step. The snapshot rows are subsequently dropped
but even if that step failed, the blobs are already permanently
unreadable.

## Opting in

Three entry points:

- **CLI**: `spark task run --forensic "<reason>" task.yaml`
- **YAML**: add a `spec.forensic` block to the task:

  ```yaml
  spec:
    forensic:
      enabled: true
      ttl_hours: 168      # 7 days
      reason: "debugging the email flow"
  ```

- **API**: the admin routes under `/api/forensic/*` expose list,
  show, snapshot, wipe, and retention-sweep operations for
  admin-role users.

## Engine integration

`spark/runtime/engine.py` instantiates a `ForensicWriter` when the
task spec's `forensic.enabled` is true and threads it through the
`_run_loop`:

- `record_prompt` is called once per iteration, right before the
  model invocation, with the assembled system prompt, user message,
  retrieved-memory context, and playbook id.
- `record_model` is called with the full model response, any
  reasoning blocks, and the requested tool calls.
- `record_tool` is called after every tool call with both the raw
  and filtered results. When a tool call raises a `SparkError`, the
  code + detail + remediation are captured.
- `finalize` runs at the very end of the run and stamps the final
  iteration and snapshot counts onto the capture row.

Every `record_*` call is wrapped in a best-effort try/except in the
engine so a forensic failure never terminates a run.

## Retention

`spark/forensic/retention.py` runs a nightly sweep (registered by
the scheduler under `spark:forensic_retention`) that calls
`ForensicRepository.wipe()` for each capture whose `expires_at` is
in the past. Default TTL is 7 days; the maximum enforced by the
Pydantic model is 90 days.

## Reading

Admin-only endpoints under `/api/forensic/*`:

| Route | Method | Purpose |
|---|---|---|
| `/api/forensic/` | GET | list captures |
| `/api/forensic/{run_id}` | GET | capture metadata |
| `/api/forensic/{run_id}/snapshots` | GET | decrypted snapshot chain |
| `/api/forensic/{run_id}` | DELETE | wipe (shred identity + drop rows) |
| `/api/forensic/retention/sweep` | POST | trigger retention sweep |

Every route writes an audit entry. The snapshot read is audited at
`info` severity; `wipe` is `elevated`.

The web UI page `/forensic` renders the list plus a detail view at
`/forensic/{run_id}` with an iteration filter, a chain sidebar, and
a payload detail pane with copy-as-JSON.

## Security properties

1. **Per-run encryption key.** Deleting the identity shreds the data.
2. **TTL enforced.** Default 7 days, nightly sweep.
3. **Admin role required for read.** Separate from operator.
4. **Every read, enable, wipe, export is audited.**
5. **Raw result AND filtered result are captured.** The filtered
   field shows exactly what the model saw; the raw shows what the
   plugin actually returned.
6. **No remote telemetry.** Local-only.
7. **Zero overhead when disabled.**

## Out of scope

- Cross-run forensic correlation (e.g. "show me every run that
  called `http_tool` with url X"). Single-run review only in v1.
- Forensic export to external SIEM. Admin-initiated local tarball
  export only — the API for it is not in v1.
- Multi-user forensic access. Admin-only, full stop.
